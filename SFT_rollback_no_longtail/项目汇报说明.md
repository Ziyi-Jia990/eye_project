# 眼底彩照阅片 SFT 项目汇报说明

## 1. 项目目标与核心问题

本项目的目标是：给定一张眼底彩照，自动生成符合临床阅片习惯的中文阅片结果，输出格式严格为两行：

```text
描述：{客观描述}
初步诊断：{可能的疾病表型类型}
```

从任务类型上看，这不是普通的图像分类，也不是纯文本生成，而是一个**医学视觉语言生成任务**。模型既要看懂图像中的眼底征象，又要把征象组织成规范的中文阅片文本，还要进一步给出初步诊断。

本项目采用的是 **SFT（Supervised Fine-Tuning，监督微调）+ LoRA** 的技术路线：  
使用已有的“眼底图像 - 阅片文本”配对数据，对多模态大模型进行有监督微调，使其学会从图像直接生成报告。

---

## 2. 项目整体流程

本项目的完整流程可以概括为 4 个阶段：

1. **读取并清洗数据**
   从 `description.csv` 中读取图像路径与对应阅片文本，过滤空报告和无法读取的坏图。

2. **按诊断标签近似分层划分数据集**
   自动生成训练集、验证集、测试集，尽量保持主要诊断类别分布一致。

3. **进行多任务 SFT/LoRA 微调**
   模型主任务是生成完整阅片文本，同时加入一个轻量诊断辅助头，帮助模型学习诊断标签信息。

4. **进行结构化生成评估**
   不只比较整段文本是否一致，而是从格式、诊断集合、病灶描述、位置、数量等级、杯盘比等多个维度评估模型输出质量。

## 3. 项目目录与关键文件

`/mnt/hdd/jiazy/eye_project/SFT` 目录下的关键文件如下：

- `train_qwen_vl_sft.py`
  训练主脚本。负责数据划分、模型加载、LoRA 注入、训练、验证、断点恢复和训练后评估。

- `evaluate_qwen_vl_sft.py`
  离线评估入口脚本。读取某个 split 和某个模型/adapter，执行生成并计算结构化指标。

- `generation_eval.py`
  真正执行生成式评估的核心代码。输出预测文件与指标汇总文件。

- `retina_sft_utils.py`
  数据读取、图像校验、分层划分、chat template 构造、训练 collator、生成 collator 等工具函数。

- `metrics.py`
  定义所有结构化评测指标，包括格式、诊断、描述、位置、数量等级和杯盘比等。

- `compare_checkpoints.py`
  用于批量比较多个 checkpoint 的结构化评测结果，帮助选择更优 checkpoint。

- `prompt.txt`
  任务提示词，规定模型的角色、描述重点和输出格式。

- `splits/`、`splits_normdiag/`
  保存训练/验证/测试集划分结果及统计摘要。

- `outputs/`
  保存训练出的 LoRA adapter、checkpoint、离线评估结果和 checkpoint 比较结果。

## 4. 数据集与监督信号

### 4.1 输入数据格式

项目当前假设 CSV 中至少包含以下字段：

- `img_id`
- `description`
- `institution_name`
- `img_path`

其中：

- `img_path` 是图像路径，项目会优先直接使用它定位图片。
- `description` 是监督目标，也就是模型要学习生成的标准阅片文本。

### 4.2 当前数据规模

根据 `splits/split_summary.json` 和 `splits_normdiag/split_summary.json`，当前项目使用的数据统计为：

- 总样本数：`55,333`
- 缺失图像数：`0`
- 空报告数：`0`
- 不同诊断组合数：`658`

数据集默认划分比例为：

- 训练集：`96%`
- 验证集：`2%`
- 测试集：`2%`

当前实际划分结果为：

- 训练集：`53,069`
- 验证集：`1,132`
- 测试集：`1,132`

### 4.3 数据划分方式

项目不是完全随机划分，而是按 `diagnosis` 字段进行**近似分层划分**。  
具体做法是：先按诊断文本分组，再在每个诊断组内按比例分配到 train / val / test。

这样做的目的有两个：

- 尽量保持不同集合中主要诊断类别分布一致；
- 减少某些少见诊断只出现在训练集、而在验证或测试集中完全消失的情况。

### 4.4 当前数据分布特点

当前数据中占比最高的类别主要包括：

- 未见明显异常表征
- 轻度近视眼底改变
- 中度近视眼底改变
- 高度近视眼底改变

这说明该数据集具有明显的长尾分布特点：常见诊断样本很多，而部分病种支持数很少，这会直接影响宏平均指标与每病种指标表现。

## 5. Prompt 设计与输出规范

`prompt.txt` 中明确要求模型扮演专业眼科医生，并遵守以下规则：

- 仅依据图像中可明确观察到的征象进行描述；
- 不臆测病史、症状或图像中不可见的信息；
- 对无异常项目不逐项写“未见异常”；
- 若图像质量差、曝光异常、遮挡或关键结构显示不清，需要明确说明；
- 输出只能包含“描述”和“初步诊断”两行，不允许输出解释、建议或额外说明。

这意味着本项目的监督目标并不是自由生成，而是**强格式约束下的医学报告生成**。

## 6. 模型方案与训练机制

### 6.1 底座模型

从项目代码和已保存 adapter 配置来看，本项目当前使用的底座模型为：

```text
Qwen/Qwen3.5-9B-Base
```

在本项目中，该模型通过 `AutoModelForImageTextToText` 加载，并作为**图像-文本到文本**的多模态生成模型使用。  
也就是说，输入是“眼底图像 + prompt”，输出是标准阅片文本。

### 6.2 为什么要用多模态底座

如果目标是“看图生成阅片结果”，底座必须具备视觉输入能力。  
纯文本模型只能处理文字，无法直接接收眼底图像，因此不能完成本项目的核心任务。

### 6.3 微调方法：LoRA

本项目没有全量微调全部参数，而是使用 **LoRA（Low-Rank Adaptation）** 对模型进行参数高效微调。  
LoRA 的优点是：

- 显著减少可训练参数量；
- 降低显存占用；
- 训练和保存更轻量；
- 便于后续切换不同 adapter 做实验对比。

### 6.4 LoRA 注入位置

根据训练脚本的默认设置，LoRA 默认注入到以下模块：

1. **语言侧投影层**
   - `q_proj`
   - `k_proj`
   - `v_proj`
   - `o_proj`
   - `gate_proj`
   - `up_proj`
   - `down_proj`

2. **视觉-语言融合层（merger）**
   - `model.visual.merger.linear_fc1`
   - `model.visual.merger.linear_fc2`

3. **视觉编码器最后若干个 block**
   默认对最后 `4` 个视觉 block 注入 LoRA，具体后缀包括：
   - `attn.qkv`
   - `attn.proj`
   - `mlp.linear_fc1`
   - `mlp.linear_fc2`

这说明本项目不是只改语言端，而是同时对**视觉侧后层、视觉-语言连接层、语言生成层**进行联合适配。

### 6.5 诊断辅助头

除了标准的文本生成损失外，项目还额外挂了一个轻量的 `diag_head`：

- 输入：诊断相关 token 的隐藏状态池化结果；
- 输出：多标签诊断 logits；
- 作用：把“初步诊断”任务显式转化为一个辅助多标签分类目标。

这样做的目的，是让模型在生成报告的同时，更稳定地学习疾病标签信息。

### 6.6 多任务损失设计

训练时的总损失由 3 部分组成：

```text
总损失 =
1.0 × 描述部分 token 交叉熵
+ 1.0 × 诊断文本 token 交叉熵
+ 0.5 × 诊断多标签 BCE 损失
```

分别对应：

- **描述 CE loss**
  只监督“描述”这一行的生成质量。

- **诊断文本 CE loss**
  监督“初步诊断”这一行的文本生成质量。

- **诊断 BCE loss**
  不直接看生成文本，而是看诊断标签的多标签分类结果是否正确。

这种设计体现了项目的核心思想：  
**既要让模型学会写报告，也要让模型学会判定诊断标签。**

### 6.7 诊断 token 池化方式

训练脚本默认使用：

```text
diag_pooling = mean
```

也就是对诊断部分 token 的隐藏状态做平均池化，再送入诊断辅助头。

## 7. 图像与文本预处理

### 7.1 图像预处理

项目会在送入 processor 前先把图像转为 RGB，并按最长边缩放。默认设置为：

```text
max_image_side = 1024
```

含义是：如果图像最长边超过 1024，则按比例缩小到 1024；否则保持原始尺寸。

这样做的主要目的是：

- 控制视觉 token 数量；
- 降低显存压力；
- 保持训练和评估时的输入尺寸一致。

### 7.2 图像质量校验

项目在训练和评估前都会先检查图片是否可正常读取。  
如果遇到损坏图片，会把信息写入 `invalid_images.json`，并自动跳过这些样本，避免训练中途崩溃。

### 7.3 文本构造方式

训练样本会被组织成 chat 格式：

- 用户输入：图像 + prompt
- 助手输出：标准阅片文本

项目还会把报告拆成：

- `描述` 部分
- `初步诊断` 部分

这样才能分别计算描述生成损失和诊断生成损失。

## 10. 评测体系设计与各指标含义

本项目没有只用整段文本 Exact Match，而是使用了**结构化评测体系**。  
这样更适合医学报告任务，因为只看整段文本是否完全一致会过于苛刻，也不能区分“格式错”“诊断错”“病灶位置错”“数量等级错”等不同问题。

### 10.1 格式指标

#### `format_correct_rate`

表示模型输出是否严格满足：

- 一共两行；
- 第一行以 `描述:` 开头；
- 第二行以 `初步诊断:` 开头。

如果某条预测满足以上条件，则记为 1，否则记为 0；  
最终对全部样本取平均。

这个指标衡量的是**输出格式服从性**。

### 10.2 诊断相关指标

首先，项目会把“初步诊断”按 `、` 切分成多标签集合，再进行比较。

#### `diagnosis_exact_set_acc`

预测诊断标签集合与参考标签集合**完全相同**时记为 1，否则记为 0。  
最终对全部样本取平均。

这个指标最严格，要求诊断既不能漏，也不能多。

#### `diagnosis_micro_f1`

把所有样本上的所有诊断标签放在一起，统计整体的：

- TP：预测和真实都存在；
- FP：预测有但真实没有；
- FN：真实有但预测没有。

再计算整体 Micro-F1。

这个指标对**高频病种**更敏感，更适合作为总体诊断能力的主指标。

#### `diagnosis_macro_f1`

先分别对每一个病种计算 F1，再对所有病种做平均。

这个指标会更加关注**少样本病种**，因此通常比 Micro-F1 更低，也更能反映长尾类别困难。

#### `diagnosis_per_disease`

对每个具体病种分别统计：

- `accuracy`
- `precision`
- `recall`
- `f1`
- `support`

其中：

- `support` 表示该病种在参考标签中出现了多少次；
- 可用于分析哪些病种容易识别，哪些病种仍然困难。

#### `diagnosis_family_level_acc`

项目把部分具有“等级/分期”关系的病种进一步归为若干家族：

- 近视分级
- 高血压视网膜病变分级
- 糖尿病视网膜病变分级
- 黄斑水肿分级
- 黄斑前膜分期

对于每个家族，只要预测出的该家族标签集合与真实集合完全一致，就记为该样本在该家族上预测正确。  
然后分别计算各家族准确率，再对这些家族准确率做平均。

这个指标主要衡量：  
**模型是否能正确区分同一疾病家族中的轻、中、重或不同分期。**

### 10.3 描述相关指标

项目不会只把“描述”当作一个纯字符串，而是会从中抽取结构化信息：

- 病灶类型（findings）
- 位置（locations）
- 数量/程度等级（bucket）
- 杯盘比（CDR）

#### `description_exact_match`

预测的“描述”文本与参考描述文本完全一致时记为 1，否则为 0。

这是最严格的描述指标。

#### `description_finding_set_f1`

把描述中抽取到的病灶类型集合做集合 F1：

- 如果病灶集合完全一致，则得分高；
- 如果漏掉病灶或多报病灶，则得分下降。

这个指标比整段 EM 更关注“**病灶有没有说对**”。

#### `description_location_f1`

把描述中出现的位置词抽取出来，例如：

- 鼻上方
- 鼻下方
- 颞上方
- 颞下方
- 上方
- 下方
- 鼻侧
- 颞侧

再对预测位置集合和参考位置集合计算 F1。

这个指标衡量“**位置是否描述正确**”。

#### `description_count_bucket_acc`

对于某些病灶，项目会进一步识别其数量/程度等级，例如：

- `散在`
- `大量`
- 默认等级（没有明确写出时记为 `default`）

如果参考文本中存在这些可量化病灶，就检查预测是否给出了相同 bucket。  
该指标只对参考中存在的 bucket 项计分。

这个指标衡量“**病灶数量级或严重程度是否匹配**”。

#### `description_cdr_mae`

对“杯盘比为 x.x”中的数值提取出来，计算预测值与参考值之间的绝对误差，然后对所有样本求平均。

MAE 越低越好。

这个指标衡量“**杯盘比数值描述是否准确**”。

#### `description_cdr_tol_acc`

如果预测杯盘比与参考杯盘比的绝对误差不超过 `0.1`，则记为命中；否则记为未命中。  
最终对所有样本求平均。

这个指标比 MAE 更直观，表示“**杯盘比是否基本落在可接受误差范围内**”。

## 11. 当前实验结果解读

### 11.1 训练阶段 best model 与结构化评估最优 checkpoint 不完全一致

当前保存的实验中，训练脚本使用的是：

```text
metric_for_best_model = eval_diag_micro_f1
```

这里的 `eval_diag_micro_f1` 来自诊断辅助头的轻量多标签分类评估，而不是完整生成后的结构化报告评估。

因此：

- 训练时选出的 best checkpoint 是 `checkpoint-3000`
- 但离线用 `compare_checkpoints.py` 做结构化生成评估时，最优 checkpoint 是 `checkpoint-13000`

这说明：

**辅助分类指标最优，并不一定等价于最终报告生成质量最优。**

对于本项目这种“最终目标是生成规范报告”的任务，后续更合理的做法，是把 best model 选择依据切换到结构化生成指标，例如：

- `eval_diagnosis_micro_f1`
- `eval_description_finding_set_f1`
- 或其他更贴合最终任务的指标

## 12. 项目特点与创新点总结

与“只做分类”或“只看整段文本是否完全匹配”的方案相比，本项目有以下几个特点：

1. **任务定义更贴近真实阅片流程**  
   直接从图像生成“描述 + 初步诊断”的规范文本，而不是只输出一个类别。

2. **采用多任务训练**  
   同时优化描述生成、诊断文本生成和诊断多标签分类，有助于提升诊断相关能力。

3. **LoRA 适配范围较完整**  
   不仅适配语言层，也适配视觉后层和视觉-语言融合层。

4. **评测体系是结构化的**  
   能把错误拆解为格式错误、病灶错误、位置错误、分级错误、CDR 数值误差等，更适合医学任务分析。

5. **具备 checkpoint 对比分析能力**  
   可以离线比较多个 checkpoint，而不是只依赖训练期单一指标。

## 14. 后续可继续优化的方向

后续如果继续推进，本项目最值得优先优化的方向包括：

1. 把训练阶段 best model 的选择指标改成结构化生成指标，而不是仅使用辅助分类头指标；
2. 针对长尾病种做重采样、损失重加权或数据增强；
3. 进一步强化输出格式约束，提高 `format_correct_rate`；
4. 对分级/分期类疾病单独设计更强的监督信号；
5. 比较不同 checkpoint、不同 prompt、不同图像分辨率对最终结构化指标的影响。
